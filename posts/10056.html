如何在 GPT 辅助下攻克杀软权限 —— 一次反客为主的经验

发表于2026-03-12|更新于2026-03-24|Security
|阅读量:

最近做梦，梦到我有一个朋友深受公司的安全软件所困扰，经常在下载或使用软件时，遇到注册表、防火墙等等各种问题。他在工位上还用着自己 2021 年机型的笔记本，恰巧这个邪恶的安全软件经常占据大量内存，感觉是最近越来越多的卡顿崩溃问题的罪魁祸首。作为一个技术纯净派，我的朋友决定控制一下安全软件的权限，我也在梦中目睹了完整的经历。为了方便，后文使用第一人称视角代替我的朋友写。

查看CA签名

首先第一步，我们需要查看相关软件 (.exe, .dll, .sys) 的签名。如果没有这个签名，我们就可以随意替换相关文件，非常简单地实施攻击。凑巧的是，我们这次的目标还真的就没有任何防护，因此我们只需要找到 .dll 中的关键入口即可随意操纵。

初步尝试

查看 dll 链接逻辑

我们可以使用 Dependencies 软件来查看 exe 启动时的依赖

比较难蚌的是这 Controller 还拼错了，草台班子这一块儿

我们先查找 exe 调用的函数，然后找到相关核心dll，再看一眼该dll中的函数。好消息是它没有做加密，我们可以很轻易地猜测各个函数的作用。

初步的想法是，让这个 start 函数直接返回 0，看看能不能骗过主进程核心策略已启动。

这是一个 32位的 dll（Machine 显示为 i386），我们使用 x32dbg 来修改汇编代码。在顶部的“符号”标签页，我们找到并双击 itxcore.dll，然后在右侧的导出函数列表中找到 Start 函数，双击后就会跳转到对应的汇编代码附近。

真是汇汇又编编啊

按下空格，将第一行替换为 xor eax, eax，表示让 EAX 寄存器值为 0，注意勾选“保持大小”和“填充NOP”

紧接着下一行输入 ret，表示执行 return，同时它会读取 EAX 寄存器的值，什么都不干直接返回 0

最后右键 $\rightarrow$ 补丁 $\rightarrow$ 修补文件，另存一下

自我保护机制

遗憾的是，当我尝试替换原本的 itxcore.dll 时，发现这个文件被锁定了，不能重命名也不能删除。这应该是杀软的自我保护机制。

我首先尝试了第三方软件 system informer，尝试直接将所有相关进程 suspend。然而，suspend 后我的文件资源管理器（Explorer）直接崩溃，我被迫重启电脑。杀软为了实现 “右键菜单扫描文件” 或者 “实时监控文件读写” 的功能，通常会把自己的一个 DLL 强行注入到系统的 explorer.exe 中，它会不断通过 IPC 通信等方式与杀软核心进程交互。当我把杀软进程 suspend 后，explorer.exe 还在等待杀软进程的回应，导致主线程被卡住，最终桌面、任务栏无响应。

没办法，还是只能进入安全模式。按住 shift 后重启电脑，选择 4 以安全模式重启电脑。这种情况下，所有的第三方驱动和杀软服务统统被微软禁止启动，我们可以直接畅通无阻地替换 DLL。

结果

再次启动，发现相关服务直接没有启动，并没有达到最初的设想，将其变为一个空壳程序。想了想，应该是砍得太前面了，程序还未来得及向操作系统注册服务，进入生命周期循环的核心逻辑。我们应该找到更深层的目标。

深入逆向

反编译

为了查看具体的函数调用逻辑，我们需要反编译相关代码，搞明白它的控制流。

反编译看了下 start，它只是个启动托盘程序（前端）的代码。我们还是在 dependencies 中寻找可以目标，最后找到真实的服务调用逻辑在 public: static int __cdecl utils::ServiceUtils::RunW(int,wchar_t * * const,class utils::IServiceCore *) 函数中。

我们使用 Ghidra 将其反编译，

在这里，我们终于找到了服务生命周期的核心代码：

这里有几个关键函数。RegisterServiceCtrlHandlerA 是大总管，把 “停止服务”、“要求暂停” 或者 “系统正在关机” 等等信号传给 FUN_100120e0 来处理，这里面应该是一堆 switch-case。我们不关心这一项。

SetServiceStatus 是程序向操作系统SCM（服务控制管理器）汇报自己当前状态。param_1 是类的 this 指针；param_1 + 7 应该是结构体 SERVICE_STATUS 类；然后 param_1[8] 是结构体里的 dwCurrentState 字段，定义在windows api中：

先是状态 2，表示服务正准备启动；然后是状态 4，表示服务已经开始运行；随后执行核心逻辑，会在这里面不断循环。最后如果核心逻辑退出，则再传入状态 1，表示服务停止。

核心逻辑 (**(code **)(*param_1 + 0x10))(); 中，*param_1 指向的是类开头的虚函数表，加 0x10 表示调用第 16/4 + 1 = 5 个虚函数。

核心调度 FUN_10012080

这里的逻辑比较简单。首先调用了类在 0x40 偏移处的成员变量的第 8/4 +1 = 3 个虚函数。很容易就能知道，这个就是干活的核心函数。拉起防御策略等等功能肯定都在里面。

如果成功运行，那就进入 WaitForSingleObject 的逻辑，等待退出信号之类的，再调用 0xc / 4 + 1 = 4 个虚函数执行清理。

理论上，我们直接把这个核心函数 NOP 掉就可以了，直接让 iVar1 = 0，骗程序核心服务已经启动。以及为了防止后续的清理函数由于空指针等问题崩溃，我们还是要把清理函数也干掉。
