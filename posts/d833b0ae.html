记录一下对Verl框架的学习

顶层设计

由于PPO系列算法涉及多个模型，因此模型的调度管理对性能有很大的影响。

Verl的做法是在每个GPU上部署一个WorkerDict类，该类拥有所有model的方法，在不同阶段可以通过bind不同的方法，摇身一变成不同的role

我还不是很确定，但我粗看代码是这样的，只能说我没有仔细看

数据流

Verl定义了一个数据传输协议，规范了不同Actor之间的数据传输，主要就是能序列化与反序列化数据，减少这里的时间瓶颈

可以使用 __getstate__ 和 __setstate__ 两个魔法方法来处理序列化操作，还是挺新颖的，第一次见，学到了。

目前我还完全没搞清出它的数据分发策略，这一部分被封装了，具体怎么分发的还需要看源码

debug

Verl基于Ray集群调度，而Ray集群重载了默认的debugger，如何在Ray集群中debug成了一件非常麻烦的事。

好在Ray有一个自己的distributed debugger，需要在VScode里装一下extensions，然后用如下代码打上断点。

有时，需要打断点的地方是并行的多个Ray Task，需要指定一下rank=0的时候打断点，查看一下上下文有没有存rank或者直接获取。

PPO 流程

以单机八卡为例，整个流程如下：

Actor Model Rollout -> Actor/Ref Model compute log prob -> Critic compute values -> calculate reward ->

Actor Model Rollout

backend默认使用vllm，最新的更新中verl已经集成了sglang，还没迁移（sglang已经薄杀vllm力，特别是并发小的时候）

只要vllm版本在0.6.3以上，就会默认调用spmd mode，不懂。

每个vllm worker都会分到 TRAIN_BATCH_SIZE/num_workers 条，num_workers 就是如果你开了tp4，那就有 $8/4=2$ 个worker。在显存可接受的范围内，应该尽量多roll？

数据会all reduce到一个worker的所有卡上，如上设定中一个vllm_worker的四张卡上都会保有roll出来的数据的副本。

code related

目前的版本中0.2.0中，vllm的cache_engine还没有集成。这里的vllm_version没有导入是None

官方留了一个传Sampling Params的位置。这里应该是为了允许用户在不同step设置不同的sampling params。

想起来有研究说，rollout时开头的token要temperature开大一点，这样多样性强一点；就可以改这里。

Actor/Ref compute log prob

rollout模型计算log prob，这里默认使用Sequence Parallel，每个worker分到 TRAIN_BATCH_SIZE/workers 条。

Sequence Parallel不熟。设为1的话就不开，之后学习Sequence Parallel的时候再仔细看看。

参数对照表

custom_reward_function.path

传入一个.py文件，里面有自定义的custom_reward_function。具体调用这个参数的位置在

custom_reward_function.name

custom_reward_function.path 对应的python文件中的一个函数名，参数列表需要和 workers/reward_manager 中你使用的reward_manager中的调用对齐。

一个例子：

trainer.val_before_train

是否在训练前先evaluate一次。
